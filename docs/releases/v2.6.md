# Version 2.6 - Release Report

## Objective
Standardize application dates and timezones, introducing timezone-independent database mappings and a unified, localized frontend format helper in Indian Standard Time (`Asia/Kolkata`).

---

## Features Implemented
- **Standard Timezone Parsing**: Configured the backend connection pool to parse dates in UTC (`timezone: 'Z'`) to maintain date consistency across local development and AWS EC2 runtime environments.
- **Dynamic JavaScript Parameters**: Parameterized timestamps in donation and comment database writes using `new Date()`, ensuring that database logs match block/signature calculations with millisecond accuracy.
- **IST Date Helper**: Created `frontend/utils/date.js` containing `formatTimestamp` which enforces the standard `Asia/Kolkata` time zone.
- **Unified Formatting**: Applied `formatTimestamp` to donor contribution histories and details page comment feeds, converting date strings into `D MMM YYYY, hh:mm am/pm` (e.g. `6 Jul 2026, 03:50 am`).

---

## Problems Found
1. timestamps were recorded using database-level `NOW()` while in-memory hashes were calculated using JavaScript's `new Date()`. This introduced a 1-second clock drift bug that could cause transaction verification hashes to mismatch.
2. The MySQL connection driver parsed date values using the Node server's local timezone. A server running in AWS EC2 (UTC) would display different times compared to local developer runtimes.
3. Comments listings were empty and failing to render because they were querying `comments.created_at` which does not exist in the database comments schema (the column is named `commented_at`).

---

## Root Cause Analysis
- **Database/Javascript Clock Offset**: Using `NOW()` executes at query evaluation time on the DB engine, which can slightly drift from the JavaScript runtime block allocation timestamp, causing floor/millisecond division mismatches.
- **Timezone Shift**: Node `mysql2` default parsing maps date inputs directly to local process hours instead of isolating them in UTC.
- **Schema Drift**: In the original database schema, the comment timestamp column was named `commented_at` instead of `created_at`.

---

## Solution
- Configured `timezone: 'Z'` in [db/index.js](file:///d:/CampusChain/backend/db/index.js).
- Parameterized date attributes using JavaScript Date objects in [donation.service.js](file:///d:/CampusChain/backend/services/donation.service.js) and [comment.controller.js](file:///d:/CampusChain/backend/controllers/comment.controller.js).
- Modified the comment insert and get queries to target the correct database column `commented_at` and aliased comments select columns to `created_at` to remain backward-compatible with frontend objects.
- Introduced `formatTimestamp` utility in [date.js](file:///d:/CampusChain/frontend/utils/date.js) and wired it into [donor-dashboard.js](file:///d:/CampusChain/frontend/donor-dashboard.js) and [fundraiser-detail.js](file:///d:/CampusChain/frontend/fundraiser-detail.js).
