# Version 2.4 - Release Report

## Objective
Standardize currency presentation, totals, and calculations for both MetaMask (ETH) and Razorpay (INR) donations across all client dashboards and campaign views.

---

## Features Implemented
- **Standardized Currency Display**: All fundraiser goal and raised metrics now present their native ETH amounts alongside local currency conversions (`~₹INR` equivalents).
- **Separated Dashboard Totals**: Updated the donor dashboard's total contributions metric to calculate and display both currencies independently, avoiding mathematical errors in mixed-currency additions.
- **Accurate Progress Calculation**: Resolved missing progress data on the NGO dashboard card list by updating backend aggregation queries.
- **CamelCase Compatibility**: Mapped snake_case database columns to camelCase aliases (`fundraiserId`, `ownerWallet`, `peopleAffected`, etc.) in the backend query return payloads to match client scripts.

---

## Problems Found
1. In `ngo-dashboard.js`, the goal was hardcoded to display as `₹${goal}`, but since goals are stored in ETH, a goal of `5 ETH` was rendered as `₹5`.
2. The NGO dashboard cards did not display the amount raised, and the progress bar remained stuck at 0% because the `getMyFundraisers` backend query did not return a `raised` column.
3. In `donor-dashboard.js`, MetaMask (ETH) and Razorpay (INR) donations were added directly without currency conversion (e.g. `0.05 + 3000` = `3000.05`) and labeled with a static `₹` symbol.
4. On the Campaign Details page, `STUDENTS IMPACTED` was rendered as `undefined` because the database field `people_affected` (snake_case) was not aliased to `peopleAffected` (camelCase) in the `getFundraiserById` query response.

---

## Root Cause Analysis
- **Missing Aliasing**: Standard MySQL SELECT `f.*` queries returned snake_case keys which standard ESM controllers passed down as-is, while the frontend dashboard models relied strictly on camelCase keys.
- **Hardcoded Formatting**: Dashboard templates hardcoded currency labels (such as `₹` prefix or `ETH` suffix) directly within the static HTML markup, rather than dynamically applying currency symbols depending on transaction metadata.
- **Lack of Currency Isolation**: The sum total calculations on the donor dashboard iterated over the array of raw `amount` values without checking the corresponding `currency` metadata field.

---

## Solution
- Rewrote backend database queries in [fundraiser.controller.js](file:///d:/CampusChain/backend/controllers/fundraiser.controller.js) and [donation.controller.js](file:///d:/CampusChain/backend/controllers/donation.controller.js) to calculate aggregate raised amounts, map return fields to camelCase, and join tables to retrieve fundraiser titles.
- Modified frontend display components in [fundraiser.js](file:///d:/CampusChain/frontend/fundraiser.js), [fundraiser-detail.js](file:///d:/CampusChain/frontend/fundraiser-detail.js), [ngo-dashboard.js](file:///d:/CampusChain/frontend/ngo-dashboard.js), and [donor-dashboard.js](file:///d:/CampusChain/frontend/donor-dashboard.js) to format figures, compute INR equivalents, and output custom currency markers dynamically.

## API Changes
- `/api/my-fundraisers`: Now returns an additional camelCase property `raised` (decimal representing amount raised in canonical ETH units) along with camelCase mappings for fundraiser profiles.
- `/api/my-donations`: Now joins with the `fundraisers` table to supply the `title` attribute for each contribution card.
